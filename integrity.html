Vorschlag: Unterstützung von Hash-Angaben im URL-Fragment und Erweiterung des integrity-Attributs um hexadezimale Hashwerte

Motivation

Beim Verlinken von Dateien oder Ressourcen im Web wäre es sehr nützlich, einen Hash im URL-Fragment (Teil nach dem # in der URL) angeben zu können, um Änderungen an der Datei leicht erkennbar zu machen. Dies ermöglicht Clients, z. B. Browsern oder Skripten, schnell zu überprüfen, ob sich eine verlinkte Datei geändert hat, ohne die Datei vollständig herunterladen oder komplexe Prüfmechanismen implementieren zu müssen.

Derzeit gibt es keine etablierte Konvention oder Standard, um im URL-Fragment einen Hashwert anzugeben. Eine mögliche, leicht lesbare Syntax wäre: 

file.txt#sha256=abcdef1234567890...

oder mit Komponente(n):

file.txt#hash=sha256:abcdef1234567890...,datum=2025-09-19

Die Einführung einer solchen Konvention könnte die Zuverlässigkeit von Downloads verbessern und Entwicklern eine einfache Möglichkeit geben, Integritätsprüfungen durchzuführen oder zu erleichtern.

Vorschlag zur Erweiterung des integrity-Attributs

Parallel zur Verwendung von Hashes im URL-Fragment sollte das HTML-Attribut integrity um folgende Aspekte erweitert werden:

  • Unterstützung von hexadezimal kodierten Hashwerten neben der bisher definierten Base64-Kodierung.
  • Klare Syntax zur Angabe des Algorithmus und der Kodierung, z. B.:
integrity="sha256:hex:abcdef1234567890..."
  • Die Kodierung (Base64 oder hex) wird explizit angegeben, um Verwechslungen zu vermeiden.
  • Optional können weitere Metadaten oder Attribute ergänzt werden (z. B. Erstellungsdatum).

Vorteile

  • Verbesserte Interoperabilität und Nachvollziehbarkeit von Datei-Links.
  • Einfachere Implementierung von Clients und Browsern, die auf Hashes im URL-Fragment reagieren.
  • Einheitliche und erweiterbare Syntax sowohl im URL-Fragment als auch im integrity-Attribut.
  • Erhöhte Sicherheit durch klare Angabe von Algorithmus und Kodierung.

Zusammenfassung

Wir schlagen vor, eine standardisierte Syntax für Hash-Angaben im URL-Fragment (Teil nach dem # in der URL) einzuführen und gleichzeitig das integrity-Attribut zu erweitern, um hexadezimale Hashwerte mit expliziter Kodierungsangabe zu unterstützen. Dies schafft eine robuste Grundlage für Dateiintegritätsprüfungen im Web.

Vielen Dank für die Prüfung dieses Vorschlags.

Dieser Vorschlag kann als Diskussionsgrundlage in den entsprechenden Web-Standards-Foren oder Repositorien eingereicht werden.

Proposal: Support for Hash in URL Fragment and Extension of the integrity Attribute

Motivation

When linking files or resources on the web, it would be very useful to specify a hash in the URL fragment (the part after the # in the URL) to easily detect changes to the file. This allows clients, e.g. browsers or scripts, to quickly verify if the linked file has changed without downloading the entire file or implementing complex verification mechanisms.

Currently, there is no established convention or standard to specify a hash value in the URL fragment. A possible, easy-to-read syntax could be: 

file.txt#sha256=abcdef1234567890...

or with component(s):

file.txt#hash=sha256:abcdef1234567890...,date=2025-09-19

Introducing such a convention could improve download reliability and provide developers with a simple way to perform or facilitate integrity checks.

Proposal to Extend the integrity Attribute

Alongside the use of hashes in the URL fragment, the HTML integrity attribute should be extended with the following aspects:

  • Support for hexadecimal encoded hash values alongside the currently defined Base64 encoding.
  • Clear syntax for specifying the algorithm and encoding, e.g.:
integrity="sha256:hex:abcdef1234567890..."
  • The encoding (Base64 or hex) is explicitly stated to avoid confusion.
  • Optionally, additional metadata or attributes can be included (e.g. creation date).

Benefits

  • Improved interoperability and traceability of file links.
  • Easier implementation of clients and browsers reacting to hashes in the URL fragment.
  • Uniform and extensible syntax both in the URL fragment and the integrity attribute.
  • Increased security through explicit specification of algorithm and encoding.

Summary

We propose introducing a standardized syntax for hash specifications in the URL fragment (the part after the # in the URL) and simultaneously extending the integrity attribute to support hexadecimal hash values with explicit encoding indication. This creates a robust basis for file integrity checks on the web.

Thank you for considering this proposal.

Proposition : Support du hash dans le fragment d’URL et extension de l’attribut integrity

Motivation

Lorsqu’on lie des fichiers ou ressources sur le web, il serait très utile de spécifier un hash dans le fragment d’URL (la partie après le # dans l’URL) pour détecter facilement les modifications du fichier. Cela permet aux clients, par exemple les navigateurs ou scripts, de vérifier rapidement si le fichier lié a changé, sans télécharger le fichier entier ou implémenter des mécanismes complexes de vérification.

Actuellement, il n’existe aucune convention ou norme établie pour spécifier une valeur de hash dans le fragment d’URL. Une syntaxe possible et lisible serait : 

file.txt#sha256=abcdef1234567890...

ou avec composante(s) :

file.txt#hash=sha256:abcdef1234567890...,date=2025-09-19

L’introduction d’une telle convention pourrait améliorer la fiabilité des téléchargements et fournir aux développeurs un moyen simple d’effectuer ou de faciliter des contrôles d’intégrité.

Proposition d’extension de l’attribut integrity

Parallèlement à l’utilisation des hashes dans le fragment d’URL, l’attribut HTML integrity devrait être étendu selon les points suivants :

  • Support des valeurs de hash codées en hexadécimal en plus du codage Base64 actuellement défini.
  • Syntaxe claire pour spécifier l’algorithme et le codage, par exemple :
integrity="sha256:hex:abcdef1234567890..."
  • Le codage (Base64 ou hexadécimal) est indiqué explicitement pour éviter toute confusion.
  • Optionnellement, des métadonnées ou attributs supplémentaires peuvent être ajoutés (par exemple date de création).

Avantages

  • Interopérabilité améliorée et meilleure traçabilité des liens de fichiers.
  • Implémentation plus simple des clients et navigateurs réagissant aux hashes dans le fragment d’URL.
  • Syntaxe uniforme et extensible à la fois dans le fragment d’URL et l’attribut integrity.
  • Sécurité accrue grâce à la spécification explicite de l’algorithme et du codage.

Résumé

Nous proposons d’introduire une syntaxe standardisée pour la spécification des hashes dans le fragment d’URL (partie après le # dans l’URL) et d’étendre simultanément l’attribut integrity pour supporter les valeurs de hash hexadécimales avec indication explicite du codage. Cela crée une base robuste pour les vérifications d’intégrité des fichiers sur le web.

Merci de prendre en considération cette proposition.

Cette proposition peut servir de base de discussion dans les forums ou dépôts dédiés aux standards web.
